# 調査・リサーチ共通ルール — 網羅性とハルシネーション対策

> 商品検索に限らず、本リポジトリの **すべての調査・リサーチ・比較タスク** に適用する共通ルール。
> 商品検索固有の詳細（3 ゲート検証・チャネル別 URL 形式・価格出典表・失敗パターン集 等）は
> `docs/rules-search-product.md` を参照する（本ファイルは**その上位・共通層**）。
> ゲート 1〜3（URL 検証・存在/在庫検証・条件充足検証）の定義は `docs/rules-search-product.md`
> 第 2 章に譲り、本ファイルでは再掲しない。本ファイルが新たに定義するのは
> **着手時の網羅性ゲート**・**ハルシネーション対策（出典規律）**・**納品前の独立検証ゲート（ゲート 4）**
> の 3 点であり、商品検索タスクではこれらが `docs/rules-search-product.md` のワークフローに
> 追加で適用される。

---

## 1. 網羅性ゲート（着手時・必須）

調査・リサーチ・比較タスクは、対象の一部だけを見て結論を出すと致命的な誤りにつながる
（商品比較における失敗パターン 6「偏ったブランド・一部商品のみの調査」が典型例）。
本章は、その失敗パターンを商品検索以外の一般的な調査タスクにも一般化したものである。

### 1.1 観点マトリクスの作成（調査を始める前に必ず）

調査に着手する前に、以下の 6 軸で「調べるべき観点」を列挙した **観点マトリクス** を作成し、
ユーザーに提示する（比較系タスクのブランド地図はこのマトリクスの 1 行目に相当する）：

1. **選択肢空間** — 直接候補（ブランド・製品・手段）／代替カテゴリ／「買わない・別解」の選択肢
2. **評価軸** — 価格・性能・品質・安全性・利便性・維持費・リスク/副作用（依頼文にない軸も列挙して取捨を明示）
3. **ユーザー文脈** — 対象者・利用シーン・制約条件（依頼文から抽出し、不明点は仮定を明示）
4. **チャネル・情報源の多様性** — EC5媒体/公式/実店舗/中古、一次情報（製品ページ・メーカー）と
   二次情報（第三者比較記事 2 本以上・レビュー・公的機関/専門家）を必ず混在させる
5. **時間軸** — 現行品か・後継/新モデルの有無・季節性・情報の鮮度（古いキャッシュ問題）
6. **反証視点** — デメリット・非推奨情報・リコール・偽レビュー・「この結論を覆す情報は何か」

観点マトリクスは Markdown 表で作成する（各軸に最低 3 項目を列挙する。埋まらない軸は
「該当なし＋理由」を明記し、空欄のまま放置しない）。例：

| 軸 | 検討事項（例） |
|---|---|
| 選択肢空間 | 候補A / 候補B / 候補C / 「そもそも買わない」選択肢 |
| 評価軸 | 価格 / 性能 / 安全性 / 維持費 / リスク・副作用 |
| ユーザー文脈 | 想定ユーザー像 / 利用シーン / 予算・制約条件 |
| チャネル・情報源 | 一次情報（公式・製品ページ） / 二次情報（比較記事・レビュー・専門家） |
| 時間軸 | 現行品か / 後継モデルの有無 / 情報の鮮度 |
| 反証視点 | デメリット・非推奨情報 / リコール・回収情報 / 偽レビューの疑い |

### 1.2 検索クエリの多様化（単一クエリ族での打ち切り禁止）

- 同義語・言い換え・英語/日本語・カテゴリ名・競合名で **最低 3 系統** のクエリを使う
- 「◯◯ 比較」「◯◯ おすすめ」「◯◯ 一覧」「◯◯ デメリット」「◯◯ 欠点」を系統に含める
- 最初の数件で「わかった気」にならず、**新情報が 2 クエリ連続で出なくなるまで**広げる（飽和判定）
- 肯定的な情報を集めるクエリだけで打ち切らない。反証視点（1.1 の 6 軸目）に対応するクエリ
  （「デメリット」「欠点」「リコール」「口コミ 悪い」等）を必ず 1 系統以上含める

### 1.3 抜け漏れ監査（coverage-critic）

- 本調査に入る前に、観点マトリクス（比較系ではブランド地図も）を `coverage-critic`
  サブエージェント（`.claude/agents/coverage-critic.md`）に渡し、抜け漏れ監査を受ける
- 指摘（重大）が 1 件でもあれば観点マトリクスを補強してから本調査に入る
- 監査結果（指摘と対応）はレポートの「調査範囲」節に要約を残す
- 監査の起動手順は `.claude/skills/coverage-planning/SKILL.md` を参照

### 1.4 調査範囲の明示（レポート必須要素）

- レポートに「調査した範囲」と「調査しなかった範囲（理由つき）」を必ず書く
- 「調べていないのに調べたように見える」レポートを作らない（**網羅性の偽装禁止**）
- 除外した選択肢・ブランド・情報源がある場合は、除外理由を添えて全件明記する
  （`docs/rules-search-product.md` のルール A「一次スクリーニング（除外製品一覧）」と同じ考え方を
  商品検索以外の調査にも適用する）

## 2. ハルシネーション対策（出典規律）

「調べた」と称する内容の一部が実際にはモデルの記憶（学習データ）から生成された架空の情報である
状態を **ハルシネーション** と呼ぶ。以下は、ハルシネーションを作成段階で防ぐための出典規律である。

### 2.1 事実主張の3分類とラベル

すべての事実主張は、以下の 3 分類のいずれかに該当することを明示して書く：

- **【引用】** 原文ママ＋ソース URL 必須（`>` 引用ブロックを使う）
- **【要約】** （出典: URL）必須。ソースにない属性・数値の追加禁止
- **【推定】** 「と考えられる」等のヘッジ語＋⚠️ 必須。出典なしの断定は禁止

数値（価格・%・件数・容量・成分量）は必ず **出典 URL＋確認日** とセットで書く。
出典を示せない数値は【推定】に格下げするか、削除する。

### 2.2 「知識」でなく「取得した情報」で書く

- モデルの記憶にある商品名・型番・価格・URL をそのまま書かない
  （実在しない型番の捏造がハルシネーションの典型パターンである）。
  必ず WebSearch/WebFetch で当該情報を取得してから書く
- URL は取得した実 URL のみ書く。記憶から組み立てた URL（存在確認していない URL）は禁止
- 確認できなかったことは「確認できなかった」と正直に書く（**埋めない勇気**）。
  空欄や「該当なし」に見えることを恐れて、それらしい値で埋めることが最も危険な失敗である

### 2.3 research レポートの機械検証

- 調査・リサーチ系レポートには、ファイル先頭（H1 より前を推奨）に
  `<!-- report-type: research -->` を付ける
- `scripts/check_research_report.py` が保存時（PostToolUse フック）に自動検証する
  （検証項目: 出典節の存在・Markdown リンク本数・確認日の記載・H1 直下メタデータ・禁止 URL）。
  違反は差し戻しとなる
- 比較・ランキング・スコアリング系レポート（商品検索）は `<!-- report-type: comparison -->` を使い、
  `scripts/check_comparison_report.py` の担当となる。両者は排他的で、1 ファイルに両方は付けない

## 3. レビューゲート（ゲート4・レポート単位の独立検証）

商品検索のゲート 1〜3（URL 検証・存在/在庫検証・条件充足検証）は候補単位の検証である。
本章の **ゲート 4** はレポート単位の最終検証であり、商品検索を含む全ての調査・リサーチ
レポートに適用する。

- レポート（比較・検索・リサーチとも）を確定・納品する前に、`fact-check-reviewer`
  サブエージェント（`.claude/agents/fact-check-reviewer.md`）による独立検証を必ず通す
- レビューアは作成に関与していない新しいコンテキストで、主張の抽出→出典との突合→
  ライブ検証（可能なチャネルのみ）を行い、✅/⚠️/❌ の判定表を返す
- ❌ が 1 件でもあれば修正して再レビュー。⚠️ は本文に ⚠️ 表記を反映してから納品する
- レビュー実施の記録（実施日・指摘件数・対応）をレポート末尾の「検証記録」節に 3 行以内で記す
- 作成者自身のセルフチェック（第 4 章）は必須だが、ゲート 4 の代替にはならない。
  「自分で書いたものを自分でチェックする」構造的な限界（確証バイアス）を、独立した
  レビューア（別コンテキストのサブエージェント）で補う

---

## 4. セルフチェック（作成者自身・レビュー前）

レポートを `fact-check-reviewer`（ゲート 4）に渡す前、作成者自身が以下を確認する：

- [ ] 観点マトリクスを作成し提示したか（着手時）
- [ ] coverage-critic の監査を受けたか（着手時）
- [ ] 数値・引用のすべてに出典 URL＋確認日があるか
- [ ] 記憶ベースの URL・型番・価格を 1 つも書いていないか
- [ ] 未確認事項を ⚠️/「確認できなかった」と明示したか
- [ ] 「調査しなかった範囲」を明記したか
- [ ] `<!-- report-type: ... -->` マーカーを付けたか（research / comparison）
- [ ] fact-check-reviewer のレビューを通したか（ゲート4）

1 つでも欠けていればレポートを破棄してやり直す（`docs/rules-search-product.md` §6.1 と
同じ「1 つでも欠けていればやり直し」の原則を踏襲する）。
