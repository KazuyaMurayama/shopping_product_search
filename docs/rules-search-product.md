# 商品検索ルール — 詳細仕様

> このファイルは CLAUDE.md から参照される商品検索専用ルール集です。
> 商品検索タスク実行時は本ファイルを必ず読んでから作業してください。

---

## 0. 最重要：過去失敗パターンと絶対遵守事項

過去の商品検索で繰り返し発生した失敗。再発したらレポートを破棄してやり直してください。

### 失敗パターン 1: 偽リンクの掲載（検索結果ページ URL を商品ページとして提示）

絶対 NG な URL 例：

    https://www.amazon.co.jp/s?k=...
    https://search.rakuten.co.jp/search/mall/...
    https://jp.mercari.com/search?keyword=...
    https://jp.iherb.com/search?kw=...
    https://osakado.org/search.html?...

対策：

- URL を掲載する前に、**個別商品ページ形式**であることを必ず検証する
- URL に以下の文字列を 1 文字でも含む場合は**即座に除外**：
  `search`, `?k=`, `?s=`, `?keyword=`, `?kw=`, `?q=`, `/search/`, `/s?`
- 個別商品 ID（ASIN / 楽天商品 ID / メルカリ item ID 等）が取得できない
  場合は「URL 取得不可」と記録し、候補から除外する

### 失敗パターン 2: 存在確認なしの紹介（実在する SKU を未確認で掲載）

「ブランドが実在する」と「その商品がそのチャネルに在庫ありで存在する」は
別問題として扱うこと。

対策：

- 一次情報（検索スニペットの本文・タイトル・価格・レビュー件数）で
  実在を必ず確認
- 「たぶん売ってそう」「一般的に売られている」での推測掲載は禁止
- 確認できない場合は「在庫確認不可」として候補から除外
- 最低限、以下のいずれかが取得できること：
  - 検索結果スニペット内に商品名と URL が明示されている
  - レビュー件数が表示されている
  - 価格が表示されている

### 失敗パターン 3: 条件の抜け漏れ（必須条件の未確認掲載）

対策：

- ユーザー入力から必須条件をリストアップし、漏れなく全項目チェック
- 各商品×各条件について `✅確認済み` / `⚠️要確認` / `❌不適合` を明記
- 1 つでも「要確認」「不適合」があれば**推奨 TOP3 から除外**
- 検索開始前に条件チェックリストをユーザーに提示し確認を取る

### 失敗パターン 4: 価格未報告 / 価格レンジ未比較

症状：レポートに「価格：要最新確認」「価格不明」が並び、ユーザーが
    実際の支出イメージを持てない。さらに中古・新古品・マーケットプレイス
    出品などの価格レンジ比較が漏れる。

対策：

- **価格は必須出力項目**。取得できなければ候補を ⚠️要確認に落とす
- スカウトエージェントは検索スニペット / 価格タグ / 通貨記号付き数値を
  最低 1 つは必ず取得する
- `¥XXX〜¥YYY` のレンジで良いので必ず数値を記載する
- レポート自己チェックに「TOP3 全商品に価格数値が入っているか」を追加
- 新品検索で候補を特定した後、**失敗パターン 4 の後段対策として
  必ず中古市場の並行検索を実施**（→ セクション 4 のフェーズ 2.5 参照）
- 価格は変動するため取得時刻を明記

### 失敗パターン 5: キャッシュ情報を現行情報として提示 / 在庫切れ未検出

症状：v2 で MAMORIO CARD (B08LL17T6D) を TOP3 1 位として ¥3,410 で推奨
    したが、実際の商品ページは「現在在庫切れです。再入荷予定は立って
    おりません」と表示されていた。検索スニペットが Google インデックスの
    古いキャッシュから取得されており、現行の在庫状況を反映していなかった。

根本原因：

- 検索スニペットは Google インデックスのスナップショットで、数週間〜
  数ヶ月前の情報を含む可能性がある
- スニペットに「currently unavailable」「在庫切れ」が書かれていない
  ＝ 在庫あり、ではない
- robots.txt で直接フェッチが制限されるチャネル（Amazon 等）では、
  現行情報の二次確認手段を用意していなかった

対策：

- **ゲート 2 に「在庫状況の現行確認」サブステップを追加**（セクション 2 参照）
- スカウトは候補特定後、**必ず** 商品名 + `"在庫"` / `"在庫切れ"` /
  `"currently unavailable"` / `"再入荷"` / `"取扱終了"` で追加検索し、
  否定情報がないか確認する
- 複数の情報源で価格・在庫をクロスチェック（最低 2 ソース）
  例：Amazon スニペット + kakaku.com / prtimes.jp / メーカー公式
- 商品の発売年が古い（2〜3 年以上前）場合は「後継モデル」の存在を
  必ず確認する
- 在庫状況が確認できない候補は `⚠️ 在庫要確認` として TOP3 除外
- レポートに **各候補の「在庫確認方法と情報源」を必ず記録**

### 失敗パターン 7: 価格を「リンク先と別チャネル／非ライブ情報源」から取り掲載 ★絶対再発禁止

症状（実例 2026-06 ホエイプロテイン）：Amazon(/dp/) は HTTP 503、楽天は
    ボット遮断で **ライブ商品ページ価格が取得できない** 環境であるにもかかわらず、
    取得できる「公式サイト価格・検索スニペット・サクラチェッカー価格」を、
    Amazon/楽天リンクの隣に **そのチャネルの価格として掲載** した。結果、ザプロは
    サクラチェッカー¥3,745・公式¥3,980 と表示したが、実際の Amazon 価格は ¥5,280。
    価格は配点 50% を占めるため、誤価格はランキングを直接歪めた（1位が再び入替）。

根本原因：
- 失敗パターン 4 は「価格の数値を出せ」までで、「リンク先チャネルのライブ価格か」
  「出典は何か」を要求していなかった
- スカウト出力では「¥3,980（公式サイト価格）」と出典が付いていたのに、レポート組立て時に
  出典注記を落とし、Amazon リンクの隣に置いた（チャネル帰属の取り違え）
- Amazon/楽天が取得不可という環境制約を回避するため、別ソースで「埋めた」

本環境での価格取得可否（実測で確定済み・2026-06）：
- Amazon `/dp/` 一次ページ … **直接取得不可（HTTP 503）**
- 楽天 `item.rakuten.co.jp` 一次ページ … **直接取得不可（bot 遮断）**
- Yahoo!ショッピング `store.shopping.yahoo.co.jp` … **取得可能（ライブ）**
- **kakaku.com / search.kakaku.com（価格集約サイト）… 取得可能**。Amazon/楽天の現行価格を
  間接取得できる（一次ページが落ちていても価格を拾える。ただし商品が混在するため
  対象 ASIN/バリアントとの一致確認が必須）

対策（比較・スコアリング系で必須／機械検証で強制）：
1. **価格は必ず「取得可能なソースから実取得した数値」を、出典を明記して掲載する**。
   優先順位：
   ① リンク先チャネルのライブページ（Yahoo! は可）
   ② リンク先チャネルの価格を集約サイト（kakaku 等）経由で取得 → 出典種別 `aggregator`・⚠️ 付き
   公式サイト価格・検索スニペット・サクラチェッカー価格を「そのチャネルの価格」として
   代用することは依然禁止（推測掲載の禁止）。
2. **Amazon/楽天価格を出す場合は、必ず kakaku 等の取得可能ソースから実取得する**。
   一次ページが 503/遮断でも「取得不可だから別の固定値で埋める」は禁止。集約サイトで
   対象バリアントの価格を取得し、`aggregator`＋⚠️ で掲載する。取得できなければ「取得不可」。
3. **「価格出典・在庫確認（機械検証用）」表をレポートに必ず設ける**（列: 商品 / リンク先 /
   表示価格 / 価格の出典 / 出典種別 / 確認日 / フラグ）。
   - リンク先チャネル ≠ 価格の出典チャネル → ⚠️ 必須
   - 出典種別が live 以外（official / snippet / **aggregator**）→ ⚠️ 必須
   - 確認日（YYYY-MM-DD）必須・30 日以内
4. この表は `scripts/check_comparison_report.py` が機械検証し、`.claude/settings.json` の
   PostToolUse フックで保存時に自動実行される（違反は exit 2 で差し戻し）。
5. 一次ページ価格を機械的に正確取得したい場合は、Amazon PA-API / Keepa / 楽天商品検索 API
   等の正規データソースを接続する（WebFetch の一次ページ取得は遮断されるため）。

### 失敗パターン 6: 偏ったブランド・一部商品のみの調査（網羅性の欠如）★最重要・絶対再発禁止

症状：比較・ランキング・スコアリング系のタスクで、市場に存在する多数の競合
    （10〜20 以上）を調査せず、最初に思いついた数ブランド・数商品だけを
    俎上に載せて「TOP3」「最終版」を確定してしまう。結果、本来上位に入る
    べき製品が丸ごと欠落し、ランキング自体が無意味になる。

実例（2026-06 コードレス掃除機）：当初 5〜6 製品（Shark・マキタ・日立中心）
    だけでランキングを確定したが、後から Roborock・Dreame・東芝・シャープ・
    パナソニック等を含む 14 製品規模の市場が存在することが判明。1 位が
    入れ替わる重大な誤りだった。

実例 2（2026-06 ホエイプロテイン ヨーグルト味）★本ルール追加後の再発：
    本「失敗パターン 6」を追加（2026-06-22）した **わずか 28 時間後**（2026-06-23）に、
    比較レポートを 8 製品だけで確定・コミットして再発させた。ユーザー指摘を受けて
    市場全体スキャンを実施したところ 12 製品規模となり、ザプロ・REYS・ビーレジェンド等が
    新たに浮上、**TOP1 がマイプロテイン→ザプロに入れ替わった**（症状どおり）。
    この再発は「ルールを文章で追加するだけでは守られない」ことの証拠であり、
    対策はワークフローの構造（起動条件・スコアリング前の必須成果物）に組み込むこと
    （下記の着手時ゲート・§4 参照）。

根本原因：

- 「思いついたブランドを調べる」発想で、カテゴリ全体のブランド地図を
  描かないまま調査に着手していた
- 「何件調べれば網羅と言えるか」の基準を持っていなかった
- 除外した製品を記録しないため、調べ落としに気づけなかった

対策（比較・ランキング・スコアリング系タスクでは必須）：

1. **市場全体スキャンを最優先フェーズとして先に実施**（スコアリングより前）
2. **ブランド地図の作成**：カテゴリの主要プレイヤーを体系的に列挙する
   - 国内大手 / 海外大手 / EC 専業・新興（中国勢含む）/ 量販店 PB まで網羅
   - 「自分が知っているブランド」だけで打ち切らず、`"カテゴリ名 比較"`
     `"カテゴリ名 おすすめ ランキング"` `"カテゴリ名 メーカー 一覧"` 等で
     **第三者のまとめ・比較記事を最低 2 本クロスチェック**して漏れを補う
3. **各ブランドの現行該当機種を洗い出し**、一次スクリーニング表を作る
4. **除外した製品も「除外理由つきで全件レポートに明記」**（第 2 章相当の
   除外一覧）。これにより調べ落としと意図的除外を区別できる
5. **件数の下限を自己チェック**：俎上に載せた候補が著しく少ない
   （目安として有効候補 15 件未満、または主要ブランドの抜けがある）
   場合は「網羅不足」とみなしランキングを確定しない
6. **着手前にユーザーへ調査対象ブランド一覧を提示**し、網羅性の合意を取る
   （「他にもあるのでは？」とユーザーに指摘される前に自分から提示する）

> このパターンは「正しく調べた一部」より有害である。**部分最適なランキングは
> 全体を誤らせる**。比較・ランキングを作るときは、必ず「市場全体を見たか？」
> を最初に自問すること。

## 0.5 比較・ランキング・スコアリングレポートの追加ルール

商品を点数化・順位付けして比較する形式のレポート（本リポジトリの掃除機・
自転車・ツール比較等）には、通常の検索レポートに加えて以下を必ず適用する。

### ルール A: 市場全体スキャンの義務化

- セクション 0 の「失敗パターン 6」対策を必ず実施する
- レポート冒頭付近に **「一次スクリーニング（除外製品一覧）」表を必ず掲載**し、
  除外した製品と理由を全件明記する

### ルール B: スコア比較表の商品名は購入リンクにする（リンク必須）

- 全製品スコア比較表（および TL;DR・サマリー表）の **商品名セルは必ず
  ハイパーリンク** `[商品名](URL)` にする
- リンク先の優先順位：
  **① Amazon.co.jp → ② 楽天市場 → ③ メルカリ → ④（上記になければ）メーカー公式**
- **できる限り「在庫があり購入可能な」Amazon / 楽天 / メルカリの個別商品
  ページ**を採用する。価格.com 等の価格比較サイトや検索結果 URL は
  最終手段（公式すら無い場合）に限る
- リンクは個別商品ページ形式のみ（第 0 章・第 1 章の禁止 URL パターン遵守）
- 在庫が確認できないリンクには ⚠️ を併記する

### ルール C: レポート内に変更履歴を書かない（最終版主義）

- レポートを更新するたびに「v2 からの変更」「改訂点」「v2→v3 ランキング
  変化」などの **変更履歴・差分説明をレポート本文に書かない**
- 読者に必要なのは「最終版の結論」だけ。何をどう変えたかの経緯は不要
- 具体的に本文へ書かないもの：
  - 「○○からの変更」列を持つ表
  - 「v2→v3 変化」「改訂点まとめ」等の差分セクション
  - 配点・項目を「なぜ変えたか」の説明（最終的な定義だけ書く）
  - H1 直下メタデータの「（〜で再スコア）」のような改訂注記
- 変更の記録が必要な場合は、レポート本文ではなく `reports/index.txt` や
  `file_index.md` の 1 行説明、または commit message に留める

### ルール D: スコア確定前にゲート 2・3 を全候補に適用する（在庫・条件確認の義務化）★絶対再発禁止

症状（実例 2026-06 ホエイプロテインヨーグルト味）：スコアリングを先に実施し
「スコアが高い順に TOP5」を決定した後で初めて在庫確認を行った。その結果、
VALX・アルプロンが SOLD OUT、GronG WPI が販売中止表示と判明し、ランキングが
崩壊した。「スコア確定時点で在庫が保証されていない比較レポート」は無価値。

根本原因：
- 比較レポートはスコアリングに意識が向き、ゲート 2（在庫）・ゲート 3（条件充足）が
  「後でやる作業」になり、最終的に省略・形骸化した
- 「商品名にキーワードが含まれる」を条件充足の確認とみなした（例：「ヨーグルト」
  という文字が商品名に入っているから条件充足、という不十分な確認）
- 「否定情報がなかった＝在庫あり」という誤推論（失敗パターン 5 の繰り返し）

対策（比較・ランキング・スコアリング系タスクでは必須）：

1. **スコアリングはゲート 2・3 通過後のみ実施可能**
   - ゲート 2（在庫確認）: 2 ソース以上で肯定的な在庫確認が完了した候補のみ
     スコアリング対象にする。1 ソースのみ・否定情報なしは ⚠️ 扱いでスコアリング
     対象外（参考掲載可）
   - ゲート 3（条件充足）: ユーザー指定の全必須条件を製品ページまたは一次情報で
     確認した候補のみスコアリング対象にする

2. **「商品名にキーワードが含まれる」は条件充足にならない**
   - 正しい確認: 製品ページ・公式サイト・検索スニペットでフレーバー名が
     明示されていること（例：「ヨーグルト風味」「Yogurt Flavor」等）
   - 不十分な確認: 商品名に「ヨーグルト」という文字が含まれているだけ

3. **在庫確認の最低基準（2 ソース以上の肯定確認）**
   - ソース例（以下から 2 つ以上）：
     - 公式ショップで「在庫あり」「カートに入れる」が表示されている
     - 楽天 / Amazon / Yahoo!で現行価格が取得できた（スニペット含む）
     - kakaku.com / my-best.com 等で現行販売店が掲載されている
     - メーカー公式の現行製品ページに記載がある
   - 「否定情報なし」は 0 ソースとしてカウントしない

4. **ゲート未通過の候補の扱い**
   - ⚠️ 在庫未確認: スコアリング表に参考掲載のみ可。推奨 TOP5 には含めない
   - ❌ 在庫なし確認: 除外製品一覧に掲載。スコアリング表から除外

5. **ワークフローの修正**（比較・ランキングタスク用）
   ```
   ① 市場全体スキャン（ルール A）
   ② 一次スクリーニング（在庫確認・条件確認・サクラ確認 → ゲート 2・3）
   ③ ゲート通過した候補のみスコアリング実施
   ④ レポート生成（除外製品一覧含む）
   ```
   スコアリングを先行させ、後から在庫確認するフローは禁止。

## 2. 3 段階の検証ゲート（全チャネル共通）

すべての候補商品は以下 3 ゲートを順に通過しなければレポートに掲載不可。

ゲート 1：URL 検証

- 個別商品ページ形式か？
- 禁止パターン文字列を含まないか？
- ID（ASIN 等）が有効な形式か？

ゲート 2：存在・在庫検証（強化版、失敗 5 対策）

- 一次情報で実在確認できたか？
- **在庫状況の現行確認を実施したか？**（以下のいずれか 2 つ以上で確認）
  1. 商品名 + `"在庫"` / `"再入荷"` / `"currently unavailable"` / `"取扱終了"`
     の追加検索で否定情報がないことを確認
  2. kakaku.com / prtimes.jp / メーカー公式ストアでの現行価格掲載を確認
  3. 検索スニペットの取得日付 / インデックス日時が 1 ヶ月以内であることを確認
- 在庫切れ・取り寄せ・入荷待ち・取扱終了でないか？
- レビュー件数 / 価格 / 商品名 のいずれか以上が取得できたか？
- **「スニペットに在庫切れと書かれていない」≠「在庫あり」**
  現行在庫の確証がない候補は `⚠️ 在庫要確認` として TOP3 除外

### 2.1 Amazon 商品のレビュー信頼性検証（サクラチェッカー必須）

**推奨 TOP3 に Amazon 商品が含まれる場合、サクラチェッカーでの検証を必須化。**

- URL 形式：`https://sakura-checker.jp/search/[ASIN]/`
- 各 TOP3 Amazon 候補について WebFetch で以下を取得：
  1. サクラ度（%）
  2. 総合評価（合格 / 注意 / 危険）
  3. 評価件数と評価分布
  4. 怪しい日本語評価 / ギフト券評価 / AI レビューの検出有無
  5. ブランド・ショップの信頼性
- **判定ルール**：
  - サクラ度 **30% 以上** または 総合評価「危険」 → **TOP3 除外**、繰り上げ必須
  - サクラ度 **10〜30%** または「注意」 → ⚠️ フラグ付きで掲載、理由を明記
  - サクラ度 **10% 未満** または「合格」 → ✅ 掲載可
  - 評価件数不足で分析不可 → ⚠️ 注意書きを添え掲載可
- レポートに必ず「サクラチェッカー評価」セクションを設けること

ゲート 3：条件充足検証

- ユーザー指定の必須条件を**全て**満たしているか？
- `✅` / `⚠️` / `❌` で明記したか？
- `⚠️` / `❌` を 1 つでも含む場合は推奨 TOP3 除外

ゲート 4：独立ファクトチェックレビュー（レポート単位・納品前必須）

- ゲート 1〜3 は候補単位、ゲート 4 は **レポート単位** の最終検証
- レポート草稿の完成後、`fact-check-reviewer` サブエージェントに草稿パスと依頼原文を渡し、
  独立コンテキストで検証させる（作成者自身のセルフチェックでは代替不可）
- ❌ が 1 件でもあれば修正して再レビュー。⚠️ は本文に ⚠️ 表記を反映してから納品
- レビュー実施の記録（実施日・指摘件数・対応）をレポート末尾「検証記録」節に残す
- 共通ルールの詳細は `docs/rules-research.md` §3 を参照

## 3. 評価・価格に関するデフォルト値

- 星評価閾値：**4.0 / 5.0 以上**
- レビュー件数の信頼下限：**10 件**（10 件未満は慎重扱い）
- 価格基準：**セール・クーポン適用前の通常価格**
- 送料込み / 別を必ず明示
- 価格・在庫は変動する旨を必ず注記

### 3.1 価格は必須出力項目（失敗パターン 4 対策）

- 全候補について**価格の数値を必ず記録する**。取得できない候補は
  `⚠️要確認` 扱いにし TOP3 から除外
- レンジ可：`¥3,000〜¥4,500（送料別）` など
- TOP3 には必ず「取得時点の具体的な数値」を入れる
- 検索スニペットに価格がない場合は、別のクエリ（例：
  `"商品名" 価格` / `"商品名" 円`）で再検索する
- 価格取得の試行を 2 回行っても不明なら、レポートにその旨を明記し
  「価格未確定商品として TOP3 除外」と記録

## 4. ワークフロー（標準）

### 4.0 着手時ゲート：タスク種別判定（最初に必ず実施）★失敗パターン 6 再発防止

**いかなる商品タスクも、スカウト・スコアリングに着手する前にまず種別を判定する。**
判定は自己想起ではなく、依頼文の語で機械的に行う。

- 依頼に次のいずれかの語・意図が含まれれば **「比較・ランキング・スコアリング系」** と判定する：
  「比較」「ランキング」「おすすめ」「TOP◯」「ベスト」「一番（良い/安い）」「コスパ最強」
  「◯点満点で評価」「順位」「どれがいい」「選ぶなら」
- **比較系と判定したら、最初のユーザー応答内で必ず：**
  1. 「これは比較・ランキング系タスクである」と明示的に宣言する
  2. § 0.5 ルール A・D と本 § 4.5（比較系ワークフロー）に従うと宣言する
  3. **スコアリングに着手しない**。先にブランド地図の作成に入る
- これにより「比較タスクだと気づかずスコアリングを先行させる」経路を遮断する。

### 4.5 比較・ランキング・スコアリング系タスクの専用ワークフロー（§ 4 標準より優先）

比較系と判定した場合、下記を **この順序で** 実行する。各ステップは前ステップの
**成果物が存在しないと次へ進めない**（成果物ゲート）。

```
ステップ 1【ブランド地図】市場全体スキャン（ルール A / 失敗パターン 6）
  - カテゴリの主要プレイヤーを体系的に列挙（国内大手/海外大手/EC専業・新興/PB）
  - 第三者の比較・まとめ記事を最低 2 本クロスチェックして漏れを補う
  - 有効候補の下限 15 件を目安に。主要ブランドの抜けがないか自己点検
  - ★成果物: 「ブランド地図（候補一覧）」をユーザーに提示し網羅性の合意を取る
    （「他にもあるのでは？」と指摘される前に自分から出す）
  ── この成果物が無ければステップ 3 のスコアリングに進んではならない ──

ステップ 2【一次スクリーニング】ゲート 2・3 を全候補に適用（ルール D）
  - ゲート 2（在庫）: 2 ソース以上の肯定確認。否定情報なしは 0 ソース扱い
  - ゲート 3（条件充足）: 製品ページ等の一次情報でフレーバー等を確認
    （商品名にキーワードが含まれるだけは不可）
  - ★成果物: 「一次スクリーニング表（除外製品＋除外理由を全件明記）」

ステップ 3【スコアリング】ゲート 2・3 を通過した候補のみ点数化
  - ⚠️ 在庫未確認・条件未充足の候補はスコアリング対象にしない（参考掲載のみ可）

ステップ 4【レポート生成】除外製品一覧を含めて出力（ルール B・C も適用）

ステップ 5【レビューゲート】fact-check-reviewer による独立検証（ゲート 4）を通過後に納品
```

スコアリングを先行させ、ブランド地図・在庫確認を後から行うフローは**禁止**。

### 4.1 標準ワークフロー（単発検索・非比較系）

1. ユーザー要件確認 → 条件チェックリスト提示 → ユーザー承認取得
2. チャネル別スカウトエージェントを順次実行（並行処理可だが確実性優先）
3. 各候補に 3 ゲート検証を適用
4. **フェーズ 2.5：中古市場サーチ**（中古が成立する商品カテゴリの場合、必ず実施）
   - TOP3 候補が確定した後、その具体商品名で以下を再検索
     - メルカリ（個人間中古）
     - 楽天市場内の中古出品
     - Amazon マーケットプレイス中古 / Amazon Warehouse
     - Yahoo!ショッピング / Yahoo!オークション
   - 各チャネルで価格レンジと状態を取得し、レポートに「中古相場」セクションで掲載
   - 中古が成立しないカテゴリ（サプリ、医薬品、食品、衛生用品等）は
     本フェーズをスキップし「中古対象外」と明記
5. レポート出力エージェントが表形式（プレーンテキスト）レポートを生成
6. レポートを `reports/search_YYYYMMDD_HHMM.txt` に保存
7. `reports/index.txt` に履歴追記
8. **レビューゲート（ゲート 4）**：fact-check-reviewer サブエージェントに草稿パスと
   依頼原文を渡し独立検証。❌ があれば修正して再レビュー、承認後にのみユーザーへ納品

中古対象外カテゴリ（サプリ / 医薬品 / 化粧品 / 衛生用品 / 下着 / 食品 等）
および中古相場セクションの書式詳細は `docs/used-market-rules.md` を参照。
該当カテゴリでは本フェーズをスキップし「中古対象外」と明記する。

## 5. レポート出力形式

- **Markdown 形式**（.md ファイル）
- ファイル名: `reports/search_YYYYMMDD_[商品キーワード]_v1.md`
- 見出しは `#` / `##` / `###` を使用
- 表は Markdown テーブル形式（`| 列 | 列 |` 形式）
- リンクは `[表示名](URL)` 形式
- セクション0（結論サマリー）の商品一覧表には必ず「購入リンク」列を設ける
- 雛形は `reports/search_20260608_wood_round_rod_v1.md` を参照

> 注: 中間生成物（チャネル別スカウト出力・Amazon 専属調査等）は引き続き .txt で保存可。最終統合レポートは必ず .md で保存すること。

## 6. 自己チェック（着手時ゲート ＋ 出力前チェック）

### 6.0 着手時ゲート（比較・ランキング系タスクで、スコアリング前に必須）★失敗パターン 6 再発防止

出力前チェック（§ 6.1）は報告直前に発火するため、スコアリング済み・場合により
コミット済みで手遅れになる。比較系タスクでは **着手時** に以下を先に確認する：

- このタスクは比較・ランキング系か？（§ 4.0 の語で判定）。該当ならユーザーに
  「比較系である」と宣言したか
- **スコアリングに入る前に「ブランド地図（候補一覧）」を作成し、ユーザーに
  提示したか**。提示前にスコアリングを始めていないか
- ブランド地図は主要ブランドを網羅しているか（有効候補 15 件目安・第三者記事
  2 本クロスチェック）
- 在庫確認（ゲート 2）・条件確認（ゲート 3）を **スコアリングの前に** 全候補へ
  適用したか（ルール D）

このゲートを通過していなければスコアリングに進んではならない。

### 6.1 レポート出力前の自己チェック（必須）

レポート確定前に以下を必ず確認：

- 掲載 URL に `search` `?k=` `?s=` `?keyword=` `?kw=` `?q=` を含まないか
- 各商品について実在・在庫の一次情報があるか
- 全必須条件について `✅` / `⚠️` / `❌` を明記しているか
- 推奨 TOP3 は全て `✅全充足` か
- **TOP3 全商品に価格の具体的な数値が入っているか**（失敗 4 対策）
- **TOP3 全商品の在庫状況を 2 ソース以上で確認したか**（失敗 5 対策）
- **TOP3 各商品について「在庫確認方法と情報源」を記録したか**（失敗 5 対策）
- **発売年が 2 年以上前の商品は後継モデルの有無を確認したか**（失敗 5 対策）
- **TOP3 の Amazon 商品について、サクラチェッカー検証を実施したか**（セクション 2.1 参照）
- **中古対象カテゴリなら中古相場セクションが存在するか**（フェーズ 2.5 対策）
- 価格変動の注記があるか
- 取得時刻が記載されているか
- ファイル名がタイムスタンプ付きか

比較・ランキング・スコアリング系レポートでは、さらに以下も確認（第 0 章
失敗パターン 6 / 第 0.5 章 対策）：

- **市場全体スキャンを実施し、主要ブランドを網羅したか**（思いつきの数件で
  確定していないか）。第三者の比較・まとめ記事で漏れをクロスチェックしたか
- **「一次スクリーニング（除外製品一覧）」を除外理由つきで全件掲載したか**
- **有効候補が十分な件数（目安 15 件以上）あるか / 主要ブランドの抜けがないか**
- **スコア比較表・サマリー表の商品名が購入リンク（Amazon＞楽天＞メルカリ＞公式）
  になっているか**。在庫のある個別商品ページを優先したか（ルール B）
- **変更履歴・差分説明（「○○からの変更」列、改訂セクション等）を本文に
  書いていないか**（ルール C・最終版主義）
- **ゲート 2（在庫確認）を全スコアリング候補に対して実施したか**（ルール D）
  - 在庫確認は「否定情報なし」だけでは NG。2 ソース以上の肯定確認が必要
  - 在庫未確認の候補をスコアリング表に含めていないか
- **ゲート 3（条件充足）を全スコアリング候補に対して実施したか**（ルール D）
  - 「商品名にキーワードが含まれる」は条件充足の確認として不十分
  - 製品ページ・一次情報での確認が必要
- **スコアリングはゲート 2・3 通過後に実施したか**（ルール D）
  - スコアリングを先行させ後から在庫確認するフローは禁止
- **fact-check-reviewer（ゲート 4）のレビューを通し、「検証記録」節を書いたか**
- **観点マトリクス（docs/rules-research.md §1）を着手時に作成・提示したか**
- **記憶ベースの URL・型番・価格を書いていないか（取得した実データのみか）**

1 つでも欠けていればレポートを破棄してやり直し。
