# Visual Rules — レポートMDのビジュアル自動付与

レポート・成果物MD（調査メモ／設計書／README／PR説明 等）を **新規作成・更新するたびに自動適用** する。コミットメッセージ・1〜2行の短いメモは対象外。

---

## 1. 適用トリガー

以下のいずれかに該当するMDを生成・更新する際、本ルールを毎回適用する：

- `## ` 以上の見出しが2つ以上あるレポート系MD
- 比較・手順・フロー・状態遷移を含む説明MD
- ユーザーが「レポート」「まとめ」「設計書」「調査結果」と呼ぶ成果物

短いメモ・コミットメッセージ・1見出し以下の文書は対象外。

---

## 2. 図の種類判定（最重要）

| 内容 | 採用する表現 | Mermaid 使用 |
|---|---|---|
| A vs B 比較・マトリクス | **Markdown テーブル**（2〜4カラム） | 使わない |
| ガントチャート的な時系列計画 | **Markdown テーブル**（フェーズ／開始／終了／担当） | 使わない（gantt 禁止） |
| 単純な順序並び（3〜5ステップ・分岐なし） | **プレーンテキストの番号付きリスト** | 使わない |
| 分岐・ループ・複数経路のフロー | **Mermaid flowchart** | 使う |
| 状態遷移（明確な状態が3〜7個） | **Mermaid stateDiagram-v2** | 使う |

判定に迷ったら **テーブル優先**。Mermaid は最後の手段。

---

## 3. Mermaid 最適化ルール

flowchart を使う場合は以下を必ず守る：

- **方向**: `LR` を基本とする。`TD`/`TB` は階層構造を強調する積極理由がある時のみ使用
- **ノード数**: 全体で **7以下**。`subgraph` 内は **4以下**
- **ラベル文字**: 全角 **12〜15字以内**。長い説明はノードに入れず本文に書く
- **subgraph 使用条件**: 「Before/After」「カテゴリA/B」など **2グループの対比** に限定。3グループ以上は禁止
- **比較図（A vs B）は Mermaid 禁止** → 必ず2カラム Markdown テーブルへ変換
- **右上コーナー回避**: GitHub の Mermaid ビューでズームUIが右上に出るため、重要ノードは右上に配置しない（`LR` なら右端、`TD` なら左端付近に余白を持たせる）

### NG → OK 変換例

**NG（TD・9ノード・比較をMermaidで書いた）:**
```
flowchart TD
    A[AI導入検討] --> B[戦略フレーム採用]
    B --> C[3ヶ月インタビュー]
    C --> D[200社ヒアリング完了]
    D --> E{経営会議}
    E -->|承認| F[PoC開始]
    F --> G[技術上の壁で頓挫]
    G --> H[予算消化]
    H --> I[プロジェクト終了]
```

**OK（LR・7ノード・ラベル短縮）:**
```
flowchart LR
    A[AI導入検討] --> B[戦略フレーム採用]
    B --> C[3ヶ月調査→200社HE]
    C --> D{経営会議承認}
    D --> E[PoC開始]
    E --> F[技術上の壁で頓挫]
    F --> G[予算消化・終了]
```

---

## 4. テーブル整形ルール

- 列数は **2〜4列** を基本（5列以上は分割を検討）
- ヘッダー行は必ず置く
- 「数値」「日付」列は右寄せ（`---:`）を明示
- 5列以上必要な場合は行と列を転置する（縦持ち←→横持ちの変換）

---

## 5. 自己チェック（成果物MD送信前）

1. このMDは §1 のトリガーに該当するか？ → Yes なら以下続行
2. 図にした候補は §2 のどれに該当するか？ → テーブルで足りないか再考
3. Mermaid を使うなら §3 の6項目（方向/ノード数/文字数/subgraph/A vs B禁止/右上回避）を満たすか
4. gantt を書いていないか？ → 書いていたら Markdown テーブルへ置換
5. ノード8個以上の図になっていないか？ → 分割または抽象化

不合格なら表現を差し替えてから push する。
