# Coverage Planning — 観点マトリクス作成と抜け漏れ監査

調査・リサーチ・商品検索・比較タスクの **着手時** に、スカウト・スコアリング・Web検索を
始める前に必ず実行する手順スキル。目的は `docs/rules-search-product.md` の失敗パターン6
（偏ったブランド・一部商品のみの調査）を商品検索以外の調査タスクにも一般化して防ぐこと。

詳細ルールの正典は `docs/rules-research.md` 第1章（網羅性ゲート）。本スキルはその
**実行手順** を定義する。

<HARD-GATE>
観点マトリクスを作成し、coverage-critic サブエージェントの監査で重大指摘ゼロ（または
2巡で収束・残課題明示）になるまで、本調査（Web検索によるスカウト・情報収集・スコアリング）
に着手してはならない。これは「簡単な調査だから省略してよい」対象ではない。
</HARD-GATE>

## 手順

1. **依頼文から「問い」と必須条件を抽出する**
   - 何を調べる／比較する依頼か、ユーザーが明示した必須条件（価格帯・容量・成分・
     ブランド指定 等）を箇条書きで書き出す
   - 依頼文に無い前提を補う場合は「仮定: 〜と解釈する」と明示する（無断で決め打ちしない）

2. **`docs/rules-research.md` §1.1 の 6 軸で観点マトリクスを作成する**
   - 6 軸: ①選択肢空間 ②評価軸 ③ユーザー文脈 ④チャネル・情報源の多様性 ⑤時間軸 ⑥反証視点
   - Markdown 表で作成し、**各軸 3 項目以上** を列挙する
   - 埋まらない軸があっても空欄のまま放置せず、「該当なし＋理由」を明記する
   - 例（雛形）：

     | 軸 | 検討事項 |
     |---|---|
     | 選択肢空間 | 候補A / 候補B / 候補C / 「そもそも買わない・やらない」選択肢 |
     | 評価軸 | 価格 / 性能 / 安全性 / 維持費 / リスク・副作用 |
     | ユーザー文脈 | 想定ユーザー像 / 利用シーン / 予算・制約条件 |
     | チャネル・情報源 | 一次情報（公式・製品ページ） / 二次情報（比較記事2本以上・レビュー・専門家） |
     | 時間軸 | 現行品か / 後継モデルの有無 / 情報の鮮度 |
     | 反証視点 | デメリット・非推奨情報 / リコール・回収情報 / 偽レビューの疑い |

3. **比較・ランキング系ならブランド地図もここで作成する**
   - 依頼が「比較」「ランキング」「おすすめ」「TOP◯」「ベスト」等（`docs/rules-search-product.md`
     §4.0 の語）に該当する場合、同 §4.5 ステップ1「ブランド地図（市場全体スキャン）」を
     このタイミングで作成する
   - カテゴリの主要プレイヤー（国内大手／海外大手／EC専業・新興／PB）を体系的に列挙し、
     第三者の比較・まとめ記事を最低 2 本クロスチェックする

4. **coverage-critic サブエージェントに監査を依頼する**
   - 渡すもの: 依頼原文 ＋ 観点マトリクス ＋（作成していれば）ブランド地図
   - 依頼先: `.claude/agents/coverage-critic.md`
   - coverage-critic は独自に Web 検索して「マトリクスに無い候補・観点」を突き合わせ、
     `[重大/中/軽]` ラベル付きの指摘と `判定: 承認 / 差し戻し` を返す

5. **重大指摘ゼロになるまで補強する（最大2巡）**
   - 差し戻し（重大指摘あり）の場合、指摘された観点・ブランド・情報源をマトリクスに追加し、
     再度 coverage-critic の監査を依頼する
   - 2 巡（監査→補強→再監査）で収束しない場合は、残っている重大指摘を「残課題」として
     マトリクスに明記した上で本調査へ進む（監査ループで無限に足止めしない）

6. **成果物をユーザーに提示してから本調査に入る**
   - 提示するもの: 観点マトリクス（＋ブランド地図） ＋ 監査結果サマリ（実施回数・指摘件数・対応）
   - この提示が完了するまでスカウト・スコアリングに着手しない
   - 監査結果の要約は、最終レポートの「調査範囲」節にも残す（`docs/rules-research.md` §1.3）

## アンチパターン

以下は本スキルが防ごうとしている典型的な失敗であり、やってはならない：

- **最初に思いついた 3 ブランドだけ調べる** — 観点マトリクスもブランド地図も作らずに
  着手し、後から「他にもあった」と発覚するパターン（失敗パターン6の再発形）
- **評価軸を価格だけにする** — 性能・安全性・維持費・リスク等、依頼文に明示されていない
  軸を最初から検討対象外にしてしまう
- **肯定情報だけ集める（反証視点の欠落）** — メリット・おすすめ記事だけを検索し、
  デメリット・リコール・悪い口コミを意図的または無意識に避ける
- **1 系統のクエリで打ち切る** — 最初のクエリで出た数件の結果だけで「わかった」と判断し、
  同義語・言い換え・英語クエリ・反証クエリを試さない

## 完了条件

- [ ] 観点マトリクスを 6 軸・各軸 3 項目以上で作成したか
- [ ] （比較系のみ）ブランド地図を作成したか
- [ ] coverage-critic の監査を依頼し、結果（承認 or 差し戻し＋対応）を得たか
- [ ] 重大指摘ゼロ、または 2 巡で収束せず残課題を明示した上で先へ進んだか
- [ ] マトリクス・監査結果サマリをユーザーに提示してから本調査に着手したか

完了したら本調査（スカウト・Web検索・スコアリング）に進む。商品検索タスクでは、続けて
`docs/rules-search-product.md` の標準ワークフロー（§4）またはゲート4（`docs/rules-research.md` §3）
に接続する。
